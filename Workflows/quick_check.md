# Quick Check - 快速验证工作流

> meta-6a skill 的默认工作流
>
> 用途：快速验证需求、识别问题、提供建议

## 触发条件

当用户使用以下关键词时触发：
- "验证6A" "检查6A" "6A检查" "meta-check" "验证一下"
- 或者请求验证、检查需求时

## 分析过程

AI 应该快速进行以下分析（内部思考，不输出）：

1. **AIAnalyze** - 快速扫描：
   - 核心问题是什么？
   - 信息是否完整？
   - 边界是否清晰？

2. **AIArchitect** - 初步评估：
   - 可行性如何？
   - 有明显风险吗？
   - 复杂度评估

3. **其他阶段** - 按需考虑：
   - 根据问题类型，判断是否需要其他阶段

## 输出格式

```
✅ 6A快速验证

📊 [问题识别]
- 核心问题：[一句话明确问题]
- 完整性：[完整/不完整 - 缺少什么]
- 紧急程度：[高/中/低]
- 影响范围：[描述]

🎯 [关键建议]（最多3条）
1. [最重要且可执行的建议]
2. [次重要建议]
3. [可选建议]

💡 [下一步行动]
[立即可以执行的具体行动]

⚠️ [风险提示]（如果有）
[明显的风险或需要注意的地方]
```

## 质量门控

在输出前，AI 应该自检：

- [ ] 核心问题是否明确？
- [ ] 建议是否可执行？
- [ ] 下一步是否清晰？
- [ ] 输出是否简洁（< 200词）？

## 示例

### 示例 1：需求验证

**用户输入**：
```
帮我验证一下这个需求："开发一个用户登录功能"
```

**AI 输出**：
```
✅ 6A快速验证

📊 问题识别
- 核心问题：需求描述过于简单，缺少关键细节
- 完整性：不完整 - 缺少认证方式、安全要求、边界定义
- 紧急程度：高 - 影响后续所有开发阶段
- 影响范围：认证系统、数据库、安全架构

🎯 关键建议
1. 澄清认证方式：用户名密码？OAuth？SSO？第三方登录？
2. 定义安全要求：密码加密、存储规范、会话管理、防暴力破解
3. 明确功能边界：只做登录，还是包含注册、找回密码、记住密码？

💡 下一步行动
建议先运行"6A框架"需求澄清，完成 AIAnalyze 阶段后再进入设计。

⚠️ 风险提示
直接开发可能导致后期返工，安全漏洞，或与现有架构冲突。
```

### 示例 2：设计验证

**用户输入**：
```
检查这个设计是否合理："使用 REST API，数据存 MongoDB，前端用 React"
```

**AI 输出**：
```
✅ 6A快速验证

📊 问题识别
- 核心问题：技术栈选择合理，但缺少架构细节
- 完整性：部分完整 - 缺少接口设计、数据模型、错误处理
- 紧急程度：中 - 可以开始，但需要补充细节
- 影响范围：前后端分离，API 设计

🎯 关键建议
1. 定义 API 接口规范（路径、方法、参数、响应格式）
2. 设计数据模型和 Schema（用户、订单、商品等）
3. 规划错误处理和重试策略

💡 下一步行动
进入 AIArchitect 阶段，绘制 API 架构图和数据模型图。

⚠️ 风险提示
MongoDB 适合原型和中小规模，如果是大规模或需要复杂事务，建议考虑关系数据库。
```

## 特殊处理

### 如果问题已经很完整

如果用户的需求或设计已经很完整，AI 应该：

```
✅ 6A快速验证

📊 问题识别
- 核心问题：[明确]
- 完整性：完整
- 紧急程度：低
- 影响范围：[可控]

🎯 关键建议
当前方案已经很完整，可以直接进入执行阶段。

💡 下一步行动
1. [具体行动1]
2. [具体行动2]

✅ 无明显风险
```

### 如果缺少上下文

如果用户的描述过于简短或模糊，AI 应该：

```
✅ 6A快速验证

📊 问题识别
- 核心问题：[模糊 - 需要澄清]
- 完整性：不完整 - [列出缺失的关键信息]
- 紧急程度：中
- 影响范围：[待定]

🎯 关键建议
建议先运行"6A框架"进行需求澄清。

💡 下一步行动
请提供更多上下文：
- [问题1]
- [问题2]
- [问题3]

⚠️ 风险提示
当前信息不足，无法提供有效建议。
```

## 输出原则

1. **简洁优先**：默认输出不超过 200 词
2. **结构清晰**：使用 emoji 和分段提高可读性
3. **可执行**：每条建议都应该是具体行动
4. **诚实**：不知道就说不知道，不要猜测

## 时间限制

- 分析时间：< 5秒
- 输出时间：< 5秒
- 总计：< 10秒
