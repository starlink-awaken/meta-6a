# 性能分析

**场景**：系统性能问题，需要快速定位瓶颈

**类型**：common

**用户输入**：
```
验证一下 "API 响应时间从 100ms 涨到 2s，数据库查询慢"
```

**AI 输出**：
```
✅ 6A快速验证

📊 问题识别
- 核心问题：性能退化 20 倍，数据库查询是瓶颈
- 完整性：不完整 - 缺少慢查询日志、索引状态、数据量统计
- 紧急程度：高 - 严重影响用户体验
- 影响范围：所有 API 接口

🎯 关键建议
1. 立即收集慢查询日志（pg_stat_statements）
2. 检查索引是否失效或缺失
3. 分析数据量增长是否导致查询计划变化

💡 下一步行动
1. 启用慢查询日志（threshold: 1s）
2. EXPLAIN ANALYZE 慢查询
3. 根据执行计划优化索引或重写查询

⚠️ 风险提示
不要盲目加索引！需要分析执行计划，错误的索引可能反而降低性能。
```

**价值**：
- 展示性能问题的系统化分析方法
- 适用于性能排查和数据库优化
- 强调数据驱动、先分析后优化的原则
